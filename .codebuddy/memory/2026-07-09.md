## Fixed GPU rendering (vello + wgpu, `--features gpu` / `--gpu` flag)
GPU path was panicking in `src/renderer/gpu.rs`. Three fixes applied:
1. Device limits: `wgpu::Limits::downlevel_defaults()` caps `max_storage_buffers_per_shader_stage` at 4, but vello 0.7 needs 5 → now use `adapter.limits().clone()`.
2. Target texture needed `TextureUsages::STORAGE_BINDING` (vello `render_to_texture` requirement) in addition to `RENDER_ATTACHMENT | COPY_SRC`.
3. Readback `bytes_per_row` must be 256-byte aligned (`COPY_BYTES_PER_ROW_ALIGNMENT`); pad to aligned stride and de-pad rows when copying into the tight RGBA buffer.
- Verified: `candy build examples/dot_move.tyx --gpu --format mp4` now succeeds (GPU adapter found = llvmpipe software rasterizer in this headless env).

## Fixed external (system FFmpeg) encoder mode — `src/renderer/ffmpeg.rs`
Previously ALL ffmpeg codecs failed with `Broken pipe` (ffmpeg exited early). Two root causes fixed:
1. ffmpeg muxers (mp4/mkv/webm) require **seekable output**; piping to stdout always fails ("muxer does not support non seekable output"), and MP4 `faststart` made it worse. Fix: write ffmpeg output to a unique temp file (`std::env::temp_dir()` + pid + atomic counter) and read the bytes back; keep `faststart` for mp4. Also set stdout to null.
2. HEVC/x265 rejects the alpha channel in RGBA input ("does not support alpha layer encoding"). Fix: for software lib encoders (X264/X265/H265) add `-vf format=yuv420p` to strip alpha. Hardware encoders (vaapi/videotoolbox/qsv) left untouched.
- Verified valid outputs: `--codec x264 --format mp4`, `--codec x265 --format mp4`, `--codec h265 --format mkv` all produce correct 908×510 streams.
- Note: `--codec x264 --format webm` is inherently invalid (H.264 can't go in WebM) — not a bug. WebM path is only used by self-contained AV1.
- ffmpeg 8.1.2 present on this box with libx264/libx265/libopenh264/libvpx.

## Fixed `rav1e` 0.8.1 AV1 tiling panic — `src/renderer/rav1e.rs`
- Symptom: `assertion failed: rect.y >= -(cfg.yorigin as isize)` at `tiling/plane_region.rs:170` (`PlaneRegion::from_slice`), aborts unless `catch_unwind` intercepts it (it runs on a rayon worker; candy's `catch_unwind` in `encode()` does intercept → E007 fallback to H.264).
- Root cause (confirmed via probe): the panic fires during INTER-prediction (motion estimation) for these frame geometries. A single still frame does NOT panic; any inter frame does. `Cs420` fails even single-frame; `Cs444` is required.
- Fix applied: force all-intra by setting `enc_cfg.min_key_frame_interval = 0; enc_cfg.max_key_frame_interval = 1;` (every frame = keyframe → no ME → no panic). Valid AV1 output, just no temporal compression. Keep `Cs444` + 64-multiple rounding.
- Probe matrix (separate child processes, 960×512, 30 frames): normal/tile1/lowlat → PANIC; intra → OK. So all-intra is the only working workaround short of bumping rav1e.
- Verified: `candy build examples/dot_move.tyx --format mp4` now yields `codec_name=av1` (tested at `-p 1` → 512×256; full-size 960×512 confirmed via probe). NOTE: debug-build rav1e all-intra is SLOW (46 frames at 960×512 >240s timeout); release builds are fine. Revisit when rav1e > 0.8.1 is available.
